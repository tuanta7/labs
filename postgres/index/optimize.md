# Optimizing SQL Queries

Optimizing SQL queries involves understanding how the database engine executes queries and applying techniques to improve performance. A key tool for query optimization is the execution plan, which describes the internal steps taken by the database to retrieve or manipulate data.

Most relational database management systems (RDBMS) such as [PostgreSQL](https://www.postgresql.org/docs/current/using-explain.html), MySQL, Oracle, and SQLite provide the `EXPLAIN` keyword to reveal these plans.

## 1. The EXPLAIN Command

The `EXPLAIN` command provides a detailed breakdown of how a query will be executed. It describes the chosen access paths, join strategies, use (or non-use) of indexes, and cost estimates.

```sql
-- Show the execution plan for a query (estimates only)
EXPLAIN SELECT * FROM users WHERE age > 30;

-- Execute the query and display actual run times and row counts
EXPLAIN ANALYZE SELECT * FROM users WHERE age > 30;

-- Provide a detailed, verbose plan with buffer usage in JSON format
EXPLAIN (ANALYZE, VERBOSE, BUFFERS, FORMAT JSON) SELECT * FROM products WHERE price < 100;
```

Execution plans are generated by the query planner (optimizer), which evaluates multiple strategies before selecting the most efficient one. The decision depends on factors such as:

- Table statistics (number of rows, distribution of values)
- Indexes available
- Join conditions and order
- Query complexity and filtering predicates

By analyzing execution plans, developers and database administrators can identify performance bottlenecks and determine whether indexes, rewrites, or schema changes are required.

### Key Metrics to Look For

Here is a typical line item and what the values mean (especially from EXPLAIN ANALYZE):

```sh
EXPLAIN ANALYZE SELECT * FROM tenk1 WHERE thousand IN (1, 2, 3, 4);

                                            QUERY PLAN
-------------------------------------------------------------------​----------------------------------------------
->  Bitmap Heap Scan on tenk1  (cost=9.45..73.44 rows=40 width=244) (actual time=0.009..0.019 rows=40.00 loops=1)
```

| Metric Name | Estimated Value | Actual Value | Meaning                                                                                                                                        |
| ----------- | --------------- | ------------ | ---------------------------------------------------------------------------------------------------------------------------------------------- |
| cost        | 9.45..73.44     | N/A          | Startup cost (time until the first row is returned) and total cost (time until all rows are returned), in arbitrary units. Lower is better.    |
| rows        | 40              | 40.00        | The estimated vs. actual number of rows processed by this specific node.                                                                       |
| loops       | N/A             | 1            | The number of times this plan node was executed. The actual time and actual rows values are averages per execution if loops is greater than 1. |
| actual time | N/A             | 0.009..0.019 | The actual startup time and total time for this node to run, in milliseconds.                                                                  |
| width       | 244             | N/A          | Estimated average width of rows output by this plan node (in bytes).                                                                           |

## 2. The ANALYZE Command

The `ANALYZE` command updates the statistical metadata that the query planner relies on to generate efficient execution plans. These statistics include row counts, distinct value counts, and distribution histograms. This is distinct from the `ANALYZE` option used with `EXPLAIN`.

```sql
-- Collect statistics for all tables in the database
ANALYZE;

-- Collect statistics for a specific table
ANALYZE users;

-- Collect statistics for specific columns
ANALYZE users (age, created_at);
```

Running `ANALYZE` regularly or allowing autovacuum/autostats to manage it ensures that execution plans are based on up-to-date data distribution. Without current statistics, the optimizer may select inefficient query plans.

## 3. Indexing

Indexes are auxiliary data structures that allow the database to locate rows more quickly, avoiding full table scans when possible. They function like a book’s index, pointing to relevant pages (rows) instead of requiring a complete read.

```sql
-- Create an index on the id column
CREATE INDEX user_id_index ON users (id);

-- Remove an index
DROP INDEX user_id_index;
```

Indexes are beneficial for queries with WHERE clauses, JOIN conditions, and ORDER BY operations. However, their use depends on the query planner's cost analysis. An index may be omitted if:

- The dataset is very small (a sequential scan may be faster).
- The query requests a large portion of the table (scanning is cheaper than many index lookups).
- The index does not align with the filtering or sorting conditions.

```sql
EXPLAIN ANALYZE SELECT full_name FROM account WHERE mobile = '097****731';
```

```sh
Seq Scan on account  (cost=0.00..2507.64 rows=1 width=23) (actual time=0.283..1.994 rows=1 loops=1)
  Filter: ((mobile)::text = '097****731'::text)
  Rows Removed by Filter: 956
Planning Time: 0.154 ms
Execution Time: 2.025 ms
```

```sql
EXPLAIN ANALYZE SELECT full_name FROM account WHERE account_no = '00123456'; -- with index
```

```sh
Index Scan using account_account_no_idx on account  (cost=0.40..2.62 rows=1 width=23) (actual time=0.060..0.061 rows=1 loops=1)
  Index Cond: ((account_no)::text = '00123456'::text)
Planning Time: 0.201 ms
Execution Time: 0.090 ms
```

### Trade-offs of Indexing

- **Storage overhead**: Indexes consume additional disk space.
- **Write cost**: INSERT, UPDATE, and DELETE operations become slower since indexes must be updated.
- **Over-indexing**: Too many indexes can degrade overall performance.

Beyond indexing and execution plan analysis, several techniques can further improve SQL performance:

- **Query Rewriting**: Restructure queries to make them more optimizer-friendly (e.g., replacing subqueries with joins).
- **Partitioning**: Splitting large tables into smaller, more manageable chunks.
- **Materialized Views**: Storing precomputed query results to avoid expensive recalculations.
- **Caching**: Using in-memory caches for frequently accessed results.
- **Vacuuming** (PostgreSQL): Reclaiming storage and maintaining healthy table structures.

## 4. Materialized View 

A materialized view is a database object that stores the results of a query as a physical table, unlike regular views which are virtual and re-execute the query each time they're accessed
